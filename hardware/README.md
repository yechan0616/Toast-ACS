# 하드웨어 — 게이트 유닛

2보드로 구성돼요. **Arduino Nano ESP32**(통신·초음파·판단)가 서버를 0.7초 주기로 폴링해 상태를 보고하고 개방/경보 명령을 받아요 (`api_contract.md` §4). **Arduino Uno R3**(구동부)는 Nano ESP32가 흘려주는 상태에 따라 서보·릴레이·LED·부저를 구동해요. 게이트는 QR을 읽지 않아서 카메라 판독 성능에 기대지 않아요.

서버 연결은 두 모드를 지원해요 (`config.h`의 `USE_WIFI`).

- **USB 유선 모드 (기본)** — Nano ESP32를 맥북에 USB-C로 꽂으면, 브리지 스크립트(`bridge/`)가 폴링을 로컬 서버 `/api/gate/poll`로 그대로 중계해요. **전파 환경과 완전히 무관**해서 대회장 Wi-Fi·블루투스가 혼잡해도 영향을 안 받아요. 서버 계약은 그대로예요.
- **Wi-Fi 모드** (`USE_WIFI 1`) — Nano ESP32가 서버에 HTTP(S)로 직접 접속해요. 라즈베리파이 상시 배포용이에요.

참고로 Nano ESP32는 ESP32-S3 칩이라 클래식 블루투스(SPP)는 지원하지 않아요(BLE 전용). 무선이 꼭 필요하면 Wi-Fi 모드를 쓰면 돼요.

## 배선

### 보드 간 링크

| 연결 | 비고 |
|---|---|
| Nano ESP32 D9 → 우노 D2 | 게이트 상태 링크 (단방향, 9600bps) |
| Nano ESP32 GND ↔ 우노 GND | **공통 접지 필수** — 빠지면 링크가 안 통해요 |

### Nano ESP32 쪽 — 초음파

| 핀 | 연결 |
|---|---|
| D5 | HC-SR04 Trig |
| D6 | HC-SR04 Echo — **1kΩ 직렬 + 2kΩ로 GND 분압** (Echo는 5V 출력, Nano는 3.3V 로직이에요) |
| GND | HC-SR04 GND |

HC-SR04 VCC는 **우노 5V**에서 받아요 (GND가 공통이라 문제없어요).

### 우노 쪽 — 구동부

| 핀 | 연결 |
|---|---|
| D13 | 왼쪽 서보 주황선 (신호) |
| D12 | 오른쪽 서보 주황선 (신호) |
| D11 | 릴레이 IN |
| D10 | 220Ω → 초록 LED (+) |
| D9 | 220Ω → 빨강 LED (+) |
| 5V | 서보 2개 빨강선 · 릴레이 VCC · HC-SR04 VCC |
| GND | 서보 2개 갈색 · LED (−) · 릴레이 GND |

플랩 배리어는 캐비닛 2개에 서보가 각각 들어가요(양쪽 플랩 동시 개폐). 두 서보는 반대 방향으로 회전하도록 펌웨어가 미러링해요. 전원은 Nano ESP32가 맥북 USB-C(브리지 겸용), 우노는 별도 USB(충전기·보조배터리)에서 받아요. 서보 2개·릴레이가 우노 5V를 쓰므로 Nano ESP32 전원은 안정적이에요.

## 문 개폐 동작

- 서버 폴링 응답이 `open: true`면 Nano ESP32가 열림 상태를 계산하고(3초 개방), 우노가 서보를 0°→90°로 부드럽게 돌려 차단바를 열어요.
- 닫히는 시점에 초음파가 사람을 계속 감지하면(최근 1.5초 이내) 닫힘을 보류해요 — 문 끼임을 막아요. 보류는 최대 7초까지예요.
- 링크가 1.5초간 끊기면 우노가 스스로 닫힘·경보 해제로 돌아가요 (페일세이프).
- 릴레이·초록 LED는 열림 상태를, 빨강 LED·부저는 서버 경보(`alarm`)를 따라가요.
- 서보는 **연속회전(360도) 서보 기준**으로 시간제 회전해요. 열 때는 `SERVO_SPIN_MS`(기본 0.4초)만큼 한 방향으로, 닫을 때는 반대 방향으로 돌고 멈춰요. 멈춰 있는 동안에는 신호를 끊어(detach) 미세 회전을 막아요. 보정은 `uno-actuator.ino`에서 해요 — 회전량이 90도에 못 미치면 `SERVO_SPIN_MS`를 늘리고, 방향이 반대면 `SERVO_OPEN_SPIN_DEG`(0)와 `SERVO_CLOSE_SPIN_DEG`(180)를 서로 바꿔요. **부팅 전에 차단바를 손으로 닫힘 위치에 맞춰 두세요** (우노는 현재 각도를 알 수 없어요).

## 펌웨어 굽기

1. Arduino IDE 보드 매니저에서 **"Arduino ESP32 Boards"** 를 설치하고 보드 **"Arduino Nano ESP32"** 를 선택해요. 라이브러리 매니저에서 **ArduinoJson** (7.x)도 설치해요.
2. `config.h.example`을 `config.h`로 복사해요 — USB 모드(기본)는 그대로 두면 돼요.
3. `gate-firmware/gate-firmware.ino`를 Nano ESP32에 업로드해요.
4. 보드를 **"Arduino Uno"** 로 바꿔 `uno-actuator/uno-actuator.ino`를 우노에 업로드해요 (내장 Servo·SoftwareSerial만 써서 추가 설치가 없어요). 링크선(D2)은 꽂은 채로 업로드해도 돼요.

## USB 브리지 (맥북 내부망 서버 연결)

1. **백엔드 실행** — 맥북에서 Spring 서버(:8080)를 켜요.
2. **Nano ESP32를 USB-C로 연결**해요.
3. **브리지 실행**:

   ```bash
   cd hardware/bridge
   npm install   # 최초 1회
   npm start     # Nano ESP32(2341:0070) 자동 탐지. 수동 지정: node index.mjs /dev/cu.usbmodemXXXX
   ```

   환경변수로 바꿀 수 있어요: `API_BASE`(기본 `http://localhost:8080`), `GATE_KEY`(기본 `dev-gate-key-change-me`).
4. **확인** — 브리지 콘솔에 `.`이 0.7초마다 찍히고, 개방 명령이 나갈 때 `→ 게이트 명령 전달: OPEN`이 뜨면 정상이에요. admin 대시보드의 게이트 상태가 **온라인**이 되고, 원격 개방 버튼으로 서보 개폐까지 확인해요.

폰(client)은 맥북과 같은 네트워크에서 `http://<맥IP>:3000`으로 접속해요 (폰 핫스팟에 맥북만 물리면 돼요 — **게이트는 USB 직결이라 네트워크 구성과 무관**해요). 브리지가 끊기면 서버가 5초 뒤 `GATE_OFFLINE` 경보를 올려서 대시보드에서 바로 보여요.

주의: 브리지와 Arduino IDE 시리얼 모니터는 같은 포트를 동시에 못 써요. 업로드하거나 모니터링할 때는 브리지를 잠시 꺼요.

## 알려진 TODO (시연 후 정리)

- **HuskyLens 인원 카운트**: 지금 `readPassedCount()`는 항상 0을 반환하는 스텁이에요. Nano ESP32의 남는 핀(D10/D11)에 UART를 재할당해 연결하고 실제 통과 인원을 반환하도록 구현해요. 개방 없이 감지되면 `UNAUTH_APPROACH`, 개방 1회에 2명 이상이면 `TAILGATE` 경보를 서버가 판단하니, 펌웨어는 숫자만 보내면 돼요.
- **TLS 인증서 검증** (Wi-Fi 모드): `sendPoll()`이 `secure.setInsecure()`로 서버 인증서 검증을 건너뛰어요. 시연 편의용이라, 운영 전에 Cloudflare 루트 CA 핀닝으로 바꿔야 해요.
- **문 열림 방치 감지**: 마그네틱 리드 스위치를 남는 핀에 붙이고 폴링 본문에 `doorOpen` 필드를 추가해요 (서버·계약 변경이 필요해요).
